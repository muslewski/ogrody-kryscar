# Smooth image loading via blur-up — Design

**Date:** 2026-06-03
**Scope:** A reusable blur-up image-loading treatment for the site's self-hosted garden images, applied first to the `/uslugi/[usluga]` hero. New: a blurDataURL generator script + a generated data map + a `BlurImage` component. The `/example-N` variants and other pages are untouched (they can adopt `BlurImage` later).

## Problem

The `/uslugi/[usluga]` hero renders `next/image` (`fill priority`) over a flat `bg-neutral-100` box with no placeholder and no transition: the viewer sees an empty gray box, then the photo snaps in. Two causes:

1. **Client-side navigation:** clicking a catalog card routes to the subpage via the client router; `priority` only preloads during the *initial document* load, so on client nav the hero fetch starts only when the new page renders — a visible blank window.
2. **First-view optimization + file weight:** the first request for each image pays a one-time `next/image` optimization cost, and the source JPEGs are chunky (~380–814 KB), lengthening the blank window.

## Goal

Show a tiny blurred preview of the **actual** photo *instantly* (inline base64, zero network), then cross-fade to the real optimized image — eliminating the gray-then-pop on both direct loads and client navigation. Build it as reusable infrastructure (a generated blur map keyed by image path + a wrapper component) so every garden image can adopt it, not just the one hero.

## Approved decisions

- **Technique:** blur-up via `next/image` `placeholder="blur"` + a per-image `blurDataURL`. (Chosen over a lightweight fade/dominant-color approach for the best "no empty box" result, and over source-JPEG recompression which doesn't address the pop itself.)
- **Reusable:** build a generator + a generated map + a `BlurImage` wrapper; apply to the `/uslugi` hero now. The catalog cards and future heroes can adopt `BlurImage` later.
- **Generation is a manual, committed step** (like `scripts/fetch-stock.sh` / `IMG`), not part of `build`/`check`. The generated map is committed; regenerate only when images change. Builds stay fast and deterministic.
- **`BlurImage` is a server component** so the full `BLUR_DATA` map stays server-side — only the one chosen image's tiny string is serialized into HTML (no client-bundle bloat).
- **Graceful fallback:** a missing blur entry → `placeholder="empty"` (today's behavior), so forgetting to regenerate degrades rather than breaking the build.

## Architecture

### Unit 1 — Blur generator (`scripts/gen-blur.mjs`, new)

A Node ESM script (sibling to `scripts/mind/generate.mjs`), wired as an npm script `"blur": "node scripts/gen-blur.mjs"`.

- Scans `public/img/**/*.{jpg,jpeg,png}`.
- For each file, uses `sharp` (already installed, v0.34.5) to produce a small (~16px-wide, aspect-preserving) low-quality WebP, base64-encoded into a `data:image/webp;base64,…` URL.
- Writes **Unit 2** with a `// GENERATED by scripts/gen-blur.mjs — run npm run blur. DO NOT EDIT.` header, entries sorted by key for stable diffs.
- Idempotent: same inputs → same output (so re-running without image changes yields no diff).

### Unit 2 — Generated map (`src/lib/blur-data.ts`, new, committed)

```ts
// GENERATED by scripts/gen-blur.mjs — run `npm run blur`. DO NOT EDIT.
export const BLUR_DATA: Record<string, string> = {
  "/img/garden/lawnTexture.jpg": "data:image/webp;base64,…",
  // … one entry per scanned image, keyed by its public path
};
```

Keyed by **public path** so any caller holding a path (e.g. `svc.img`, which is `"/img/garden/…"`) can look it up directly. ~24–28 entries, ~20 KB total.

### Unit 3 — `BlurImage` component (`src/components/BlurImage.tsx`, new, server component)

A thin wrapper over `next/image`:

```ts
export function BlurImage({
  src, alt, fill, width, height, sizes, className, priority,
}: {
  src: string;
  alt: string;
  fill?: boolean;
  width?: number;
  height?: number;
  sizes?: string;
  className?: string;
  priority?: boolean;
}) { /* … */ }
```

- Looks up `BLUR_DATA[src]`. If present → render `next/image` with `placeholder="blur"` + `blurDataURL`. If absent → `placeholder="empty"`.
- Passes through `fill`/`width`/`height`/`sizes`/`className`/`priority` to `next/image`.
- No `"use client"` — works as a Server Component; the map import stays server-side.

### Unit 4 — Apply to the `/uslugi/[usluga]` hero (`src/app/uslugi/[usluga]/page.tsx`, edit)

Replace the raw `<Image>` in the hero with `<BlurImage src={svc.img} alt={svc.title} fill priority sizes="(min-width: 1280px) 1280px, 100vw" className="object-cover" />`. Keep the sized `relative aspect-[16/9]` parent and the `bg-neutral-100` as ultimate fallback. Drop the now-unused direct `next/image` import from this file if `BlurImage` fully replaces it.

## Data flow

`gen-blur.mjs` (manual) → `blur-data.ts` (committed) → `BlurImage` (server) reads `BLUR_DATA` by path → emits `next/image` with an inline blur placeholder → browser paints the blur instantly, then cross-fades to the real avif/webp-optimized image.

## Error handling

- Image path with no blur entry → `placeholder="empty"`; image still renders (graceful degrade).
- `sharp` failure on a single file in the generator → log and skip that file (don't abort the whole map); the missing entry degrades per above.
- `BlurImage` requires `alt` (accessibility); `fill` XOR `width`/`height` follows `next/image`'s own rules (TypeScript + next/image runtime enforce).

## Testing / verification

No test runner in this repo. Gate = `npm run check` (tsc + eslint + mind generator). Verify via:
- `npm run blur` regenerates `src/lib/blur-data.ts` (24+ entries); re-running produces no diff (idempotent).
- `npx tsc --noEmit` + `npx eslint` clean on the new/edited files (generated map is valid, lint-clean TS).
- `npx next build` succeeds; `/uslugi/[usluga]` still prerenders all 8.
- `next start` smoke: a `/uslugi/*` page's HTML embeds a `data:image/webp;base64,…` blur placeholder on the hero `<img>`; visually the gray-then-pop is replaced by blur-then-sharpen (and on client navigation from a catalog card).

## Out of scope (YAGNI for v1)

- Adopting blur in the catalog cards' `WarpedHoverImage` (plain `<img>` + WebGL warp — trickier integration; clean follow-up now that the infra exists).
- Source-JPEG recompression / responsive `srcset` tuning beyond what `next/image` already does.
- Running the blur generator automatically in `build`/`check` (kept manual + committed; graceful fallback makes a forgotten regen non-fatal).
- Blur for the team portraits / non-`/img` assets (the generator scans all of `public/img/**`, so they're covered if present, but no page wiring is in scope).
