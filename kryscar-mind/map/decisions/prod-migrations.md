## Context
The Postgres adapter auto-pushes the schema in dev (fast, no migration churn), but
push in production is unsafe (ambiguous diffs can drop columns/data) and was disabled
by default — leaving prod with no way to get its schema. The team-schedule MVP added a
new `visits` table + a widened `service_requests.status` enum + `declineReason`, making
a real migration story a hard prerequisite for shipping.

## Decision
- `postgresAdapter({ push: process.env.NODE_ENV !== "production" })` — dev pushes,
  prod never does and relies on `src/migrations`.
- A baseline migration (`src/migrations/20260610_171222.ts`) holds the entire current
  schema (all collections + Payload system tables), generated by `payload migrate:create`.
- `package.json`: `migrate` / `migrate:create` / `migrate:status`; the prod build is
  `payload migrate && next build` so Vercel applies pending migrations before building.

## Why
Build-time `payload migrate` keeps deploys single-step (merge → Vercel build migrates +
builds) without a separate ops task, and push:false makes prod schema changes explicit
and reviewable. The build environment already has `DATABASE_URI`, and Neon is reachable
from the Vercel builder.

## Future-change workflow
1. Edit collections; dev push reconciles your local DB automatically.
2. `npm run migrate:create -- --name <change>` → commit the generated `src/migrations/*`.
3. Merge → the Vercel build runs `payload migrate` (applies it) then `next build`.

## First-deploy runbook (one-time)
Prod has no `payload_migrations` ledger and may hold dev-pushed tables. Once:
1. Confirm no real prod data (team confirmed). Drop the prod `public` schema
   (`DROP SCHEMA public CASCADE; CREATE SCHEMA public;`) **or** use a fresh Neon branch.
2. Set `DATABASE_URI` (+ `PAYLOAD_SECRET`, `BETTER_AUTH_SECRET`, `BLOB_READ_WRITE_TOKEN`,
   the Google/Mapbox keys) in Vercel prod env.
3. Deploy — the build's `payload migrate` baselines the schema; then seed
   (`tsx --env-file=.env scripts/seed.ts && scripts/seed-services.ts`).

## Caveats
- `payload migrate` in the build runs for EVERY deployment env (preview + prod) against
  whatever `DATABASE_URI` that env carries — keep preview/prod DBs separate (or previews
  pointed at a disposable branch DB) so a preview never migrates prod.
- Migrations are forward-only here; a failed `migrate` fails the build (deploy aborts)
  before `next build`, which is the safe ordering.
